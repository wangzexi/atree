# atree 未来想法池

本文档记录尚未 trade-off 的想法。

这里的内容不代表近期要做，也不代表已经进入 MVP。它只是把讨论中出现过的方向先归档，后续可以从这里挑选、拆解、评估成本和收益。

## 设计原则

- 不把未来想法提前塞进 MVP。
- 不因为想法有趣就默认实现。
- 先保留语义，再做产品取舍。
- 进入实现前必须重新评估复杂度、用户价值和依赖条件。

## 1. Interface Session

曾经讨论过“自治目录”和“接口 Agent”：

```text
普通目录：父级 Agent 可以透明读取
自治目录：父级 Agent 默认通过接口 Agent 访问
```

在当前收束后的模型里，Agent 不是一级对象，会话才是 Agent 的表现形式。因此未来如果实现，应该叫：

```text
interface session
```

可能语义：

- 一个目录最多有一个 interface session。
- 父级访问该目录时，优先询问 interface session。
- interface session 负责解释目录状态、暴露能力、汇报风险。

暂不进入 MVP，因为它会引入父子目录访问拦截、权限边界和跨目录通信协议。

## 2. 自治目录访问边界

未来可以重新引入目录边界：

```text
默认：父级会话可读子目录
例外：子目录存在 interface session 后形成边界
```

用途：

- 控制上下文扩散。
- 让归档目录、项目目录、外部挂载目录有自己的解释权。
- 避免上层 Agent 无序读取大量内部材料。

风险：

- 增加权限模型复杂度。
- 增加调试成本。
- 需要清楚区分正常访问、管理员穿透、迁移、审计。

## 3. 文件树挂载能力

atree 的长期方向仍然是统一信息树。

未来可挂载：

- 本机目录。
- 远程电脑目录。
- Git 仓库。
- S3 / 对象存储。
- 网盘。
- 聊天软件。
- 邮件。
- 微博、Bilibili、小红书、抖音等平台。
- API 和数据库导出。

MVP 暂不实现外部平台挂载。第一阶段只处理本地根目录。

## 4. 多平台内容分发

讨论过自媒体分发场景：

```text
/内容生产
  /Bilibili
  /抖音
  /小红书
  /微博
```

父目录会话检测到新素材后，把任务分发到各平台子目录。

未来可能形成：

- 父会话做素材识别和分发。
- 平台子目录会话负责改写、发布、检查状态。
- 子目录向父目录显式汇报。

暂不进入 MVP，因为需要平台 API、账号权限、发布确认、安全策略。

## 5. React 式信息流

曾经用 React 组件树类比目录信息流：

```text
父 -> 子：下发上下文、任务、素材、约束
子 -> 父：显式 report / event / request
```

这个方向仍有价值。它能避免复杂的多 Agent 对话系统，把通信落到文件树和事件上。

未来如果实现，应该优先使用文件/事件协议，而不是让 Agent 互相自由聊天。

## 6. 图结构和跨目录引用

当前主结构坚持树。

未来可能需要图关系：

- 一个文件关联多个项目。
- 一个会话引用多个目录。
- 一个外部平台账号服务多个内容线。
- 一个任务依赖多个上下文。

原则：

```text
树是主结构
图是辅助索引
```

不要让图结构替代目录树成为主交互模型。

## 7. 可视化经营面板

长期想象中，atree 可以像“模拟经营自己所有数据”的界面：

- 目录像部门/业务线。
- 会话像员工/执行线程。
- 周期会话像自动化岗位。
- 右侧或全局面板展示整棵信息树的运行状态。

可能 UI：

- 大矩形树形面板。
- 全局浏览模式。
- 类经营游戏的可视化状态。
- 更图形化、更卡通的 Agent 形象。

暂不进入 MVP。第一版参考 Codex App，保持左侧目录树 + 右侧会话。

## 8. 全局技能和自配置能力

这个方向很重要，但需要逐步实现。

目标：

- 用户通过自然语言配置 atree。
- AI 知道如何修改 `.agents/atree.yaml`。
- AI 知道如何创建会话、设置 schedule、修改 icon/title。
- AI 知道全局配置和 API Key 应该放哪里。
- AI 知道哪些信息不能写进项目目录。

可能配置位置：

```text
~/.atree/config.yaml        # 全局配置
~/.atree/atree.sqlite       # 索引、队列、运行态
Keychain / env / secrets    # API Key
.agents/atree.yaml          # 目录级配置
```

原则：

```text
文件是语义层
SQLite 是运行层
Secrets 不进目录
```

## 9. SQLite 运行层

讨论过是否使用 SQLite。

当前倾向是混合模型：

- `.agents/` 是可读、可迁移、可被 Agent 理解的事实源。
- SQLite 是索引、事件投影、调度队列、缓存和运行锁。

适合 SQLite：

- 扫描出来的目录树索引。
- 会话列表投影。
- `next_run_at` 调度队列。
- SSE 事件投影。
- 最近打开记录。
- 搜索索引。
- 运行锁。
- 失败重试状态。

不适合只放 SQLite：

- 会话原始历史。
- 目录是否为 atree 节点。
- 用户希望跟目录一起迁移的配置。
- 技能说明。
- 附件。

## 10. Pi 生态兼容

长期希望接入 Pi Coding Agent 生态。

已经确认：

- 本地参考仓库是 `/Users/zexi/workspace/refs/pi-mono`。
- 主要 SDK 是 TypeScript。
- Pi 会话是 JSONL。
- Pi 支持 skills、extensions、ResourceLoader 等机制。

可能路线：

```text
MVP：直接使用 Pi SDK + 兼容 Pi skills 目录
v2：兼容 Pi extension discovery / ResourceLoader
v3：让 atree 目录成为 Pi extension runtime 的上下文
```

目标是尽可能接入 Pi 生态。MVP 先保证核心会话运行、JSONL、skills 与 Pi 对齐；完整扩展兼容涉及工具生命周期、权限、UI renderer、配置发现、沙箱和事件协议，需要单独评估。

## 11. 会话历史的自动模式提取

核心愿景之一：

```text
临时会话
  -> 重复使用
  -> 提取 pattern
  -> 周期会话
  -> 持续执行
```

未来可以让系统主动识别：

- 用户是否多次做类似任务。
- 是否可以总结 SOP。
- 是否可以建议转为周期会话。
- 是否可以自动生成 schedule 和执行说明。

MVP 先由用户在对话里显式要求设置 schedule。

## 12. 搜索与历史回溯

未来需要更强的历史访问能力：

- 搜索某目录所有会话。
- 搜索所有 atree 节点。
- 按时间回看周期会话执行。
- 从旧会话中提取模式。
- 根据会话历史生成目录状态。

MVP 只要求会话能保存和恢复。

## 13. 权限、安全和外部访问

应用预期作为本地 HTTP 服务运行，并支持外部设备访问。

未来需要考虑：

- 访问 token。
- 局域网访问。
- 外网反代。
- API Key 存储。
- 发布类动作二次确认。
- 删除/移动文件确认。
- 远程电脑挂载权限。

MVP 暂时只记录方向，不展开完整安全模型。

## 14. 失败恢复和运行可靠性

周期会话未来需要处理：

- 执行失败。
- 重试策略。
- 同一会话并发唤醒。
- 运行锁。
- 上次执行未完成。
- 用户中途打断。
- 失败状态如何展示。

MVP 可以先做简单串行执行，不做复杂恢复。

## 15. 目录级状态汇总

曾经讨论过节点状态，但已经收束为：

```text
状态属于会话，不属于目录
```

未来 UI 可以从会话状态中聚合出目录提示：

- 有周期会话即将运行。
- 有会话失败。
- 有会话等待用户确认。
- 最近有新会话更新。

但目录本身不引入独立状态机。

## 16. 文档和自解释

希望 atree 像 Pi Coding Agent 一样有很好的自解释文档。

未来需要：

- 完整协议文档。
- 全局技能文档。
- 配置说明。
- 扩展开发说明。
- 会话 JSONL schema。
- 附件策略。
- 调度语义。
- 示例目录。

这类文档本身也应该被 Agent 自动注入，让 Agent 能指导用户配置 atree。

## 17. 目录会话原始数据

一个目录下发生过的会话，本身就是这个目录的数据的一部分。

也就是说，目录不只包含传统文件，还包含：

- 在这个目录里启动过的临时会话。
- 长期复用的常驻会话。
- 周期执行的会话。
- 会话中的用户消息、模型回复、思考、工具调用和执行结果。
- 会话相关的标题、emoji、归档状态、定时配置等元数据。

这个判断的核心是：会话原始数据不应该被当作可丢弃日志。它记录了这个目录里真实发生过的工作、决策、尝试和执行过程。

后续可以为目录生成总结、索引、标签、目录记忆或压缩视图，但这些都应该是派生数据，而不是原始数据的替代品。原始会话记录应该默认保留；归档只是从默认视图里隐藏，不等于删除。

这个原则可以整理为：

```text
会话是目录的一等数据，而不是应用的一等数据。
```

实现上，物理存储不一定要直接放在目录文件夹内。MVP 可以继续使用全局数据库，只要每个会话有稳定的目录归属，产品语义上它就属于该目录。
